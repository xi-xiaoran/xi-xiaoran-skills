# 04_experiment_commander.skill

## skill id
`xi-xiaoran.experiment_commander`

## 一句话定义
把科研实验从“手工操作”变成“可迭代、可覆盖、可分析”的流水线工程。

## 核心特征

这个 skill 不是只会跑实验。  
它会天然去追求：

- 固定目录结构
- 可直接覆盖更新的 zip 包
- 清晰的运行命令
- 每一步明确输入输出
- 跑完之后知道该回传哪些日志
- 总结脚本一次性汇总结果
- 尽量避免人工改文件名和手工拼表

## 偏好配置

### 文件与交付习惯
- 顶层文件夹名最好固定，不要每轮改
- 之后只要解压覆盖即可
- 尽量只给必要修改文件
- 运行说明直接给 Linux 命令
- 大实验前先 smoke test

### 结果管理习惯
- 最好自动生成 summary 表
- 多环境/多方法结果最好统一汇总
- 文件名不要让人手动 rename 到崩溃
- 能 resume / skip completed steps 就更好

### 协作习惯
- 我给你代码包
- 你跑
- 你把 logs / outputs / summary 发回来
- 我分析下一步
- 不要来回反复确认已经说过的信息

## 为什么强

很多研究做不快，不是因为缺 idea，  
而是因为工程交付太乱，导致：

- 跑一次要重新想目录
- 改一次要手工抄文件
- 总结结果要自己拼
- 重跑会覆盖或混淆
- 大规模实验没人敢碰

这个 skill 就是专治这些。

## 典型输出长相

- 一个 zip 包
- 固定文件夹名
- 新增/修改文件清单
- 3～8 条可直接复制的命令
- 说明先跑什么，再跑什么
- 说明跑完把哪些文件/日志发回来

## 常见句式

- “以后每轮代码压缩包里面文件夹名字都别改。”
- “后续只要给我修改的文件我解压覆盖就行。”
- “运行都用 Linux 命令。”
- “先 smoke，再上大规模。”
- “跑完发我 summary / logs，我来分析下一步。”

## 隐含能力

### 1. 把实验目标结构化
它会要求每组实验都回答一个问题，例如：

- 是证明更快？
- 还是证明更强？
- 还是证明更稳？
- 还是证明机制预测对？

### 2. 兼顾学术与工程
不是只跑出数，而是想把这些数自然接进论文叙事里。

### 3. 默认考虑现实约束
包括：

- GPU 时间
- 并行开销
- 磁盘空间
- checkpoint 崩溃
- 下载权重困难
- 网络问题
- 远程服务器操作习惯

## 最佳使用场景

- 多阶段实验项目
- 多 seed / 多 dataset / 多 baseline matrix
- rebuttal 期间的快速补实验
- 需要反复覆盖更新代码包的协作流程
